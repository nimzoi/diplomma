# Task 03: Describe and Document Different Data Types

**Grading:** up to 10 points.

---

## 1. General principles (for all data types)

Before going into specific data types, follow these rules for everything:

1. **Consistency**

   * Use a **clear folder structure**, e.g.:

     * `data/raw/`
     * `data/processed/`
     * `data/docs/` (for README, codebooks)
     * `figures/`
     * `tables/`
     * `media/audio/`, `media/video/`, `media/images/`
   * Use **consistent file naming**, e.g.:

     * `exp01_temperature_table_raw.csv`
     * `exp01_speech_recording_01.wav`
     * `dataset_cameraspecs_image_001.jpg`

2. **Versioning**

   * Keep raw data read-only: never overwrite it.
   * Store derived data separately:

     * `data/raw/...`
     * `data/processed/...`
   * If you apply important transformations, save intermediate versions, e.g.:

     * `dataset_text_cleaned_v1.csv`
     * `dataset_text_tokenized_v2.csv`

3. **Metadata & documentation**

   * For each dataset, create a short **README** (TXT/MD) that explains:

     * What the data contains
     * How it was collected
     * Units, formats, scales
     * Any preprocessing steps
   * For structured data, create a **data dictionary / codebook**:

     * Name of each field (column or variable)
     * Type (numeric, text, categorical)
     * Meaning
     * Possible values and units

4. **Ethics and legal aspects**

   * Make sure you respect:

     * Copyright (e.g. for images, audio, video from the internet)
     * Privacy (e.g. faces, voices, personal information)
     * Any university or supervisor requirements
   * If you use external datasets:

     * Save the license or terms of use.
     * Cite the source in your thesis.

5. **Reproducibility**

   * Whenever possible, automate data preparation with scripts (Python, R, MATLAB, etc.) and store them in:

     * `scripts/` or `notebooks/`
   * Document important parameters in the thesis or in an appendix:

     * Thresholds, filters, algorithms
   * Avoid manual changes without documentation.

6. **In the thesis text**

   * Every table, figure, or media object must have:

     * A **number** and **caption** (e.g. *Figure 3.2*, *Table 4.1*)
     * A **reference** in the text:

       * “As shown in Figure 3.2…”
       * “Table 4.1 presents…”
   * Explain what the reader should notice in the data—don’t just show it.

## 2. Tabular data (tables, CSV, Excel, databases)

### 2.1 Preparing tabular data

**File formats**

* Prefer machine-readable formats:

  * `.csv` for simple tables
  * `.xlsx` if you need multiple sheets or formatting for human reading
  * `.sql` or `.parquet` if using databases / big data
* Avoid mixing formatting with data (e.g. colored cells as encoding).

**Structure**

* Each row = one observation / record / sample.
* Each column = one variable / feature.
* Avoid merged cells and multi-row headers in the raw data file:

  * Header row: short, machine-friendly column names (no spaces if possible):

    * `sensor_id`, `timestamp`, `temp_celsius`
* No “meaningful” formatting:

  * Don’t use colors or fonts to encode data values. Use columns instead.

**Data cleaning**

* Handle missing values explicitly:

  * Decide on a consistent representation: `NA`, empty cell, `-1` (only if justified).
  * Document the meaning: “NA = value not measured”.
* Check data types:

  * Convert numeric columns to numbers (not text).
  * Convert dates to proper date/time format.
* Remove obvious errors:

  * Negative distances, impossible timestamps, duplicates.
  * Document what you removed and why.

**Documentation (data dictionary)**
For each column, specify:

* Variable name: `temp_celsius`
* Description: “Measured temperature in the test chamber”
* Unit: `°C`
* Type: float / integer / categorical / boolean
* Allowed or typical range: e.g. `[-40, 120]`
* Meaning of special values:

  * `-999` = sensor error

**Example of a short codebook entry**

* `id` – record identifier, integer, unique key
* `timestamp` – measurement time, ISO 8601 (e.g. `2025-01-15T12:30:00`)
* `temp_celsius` – temperature, numeric (float), unit °C
* `status` – categorical:

  * `OK` – valid reading
  * `ERR` – sensor error
  * `MISS` – missing reading

### 2.2 Using tabular data in the thesis

**Presenting tables**

* Use tables in the thesis to show:

  * Summary statistics
  * Key parameters
  * Small subsets of data (not entire large datasets)
* Each table should have:

  * Number: **Table X.Y**
  * Title: short but clear:

    * “Table 4.1: Summary of sensor readings in experimental setup A”
* Column headers in the thesis should be human-friendly:

  * You can be more descriptive than in raw data, e.g. `Temperature [°C]` instead of `temp_celsius`.

**Text references**

* Always refer to tables in the text:

  * “Table 4.1 summarizes the temperature measurements collected during the first experiment.”
* Interpret the table:

  * Point out patterns:

    * “As shown in Table 4.1, the average temperature did not exceed 35 °C for any test case.”

**Appendices vs. main text**

* Large tables (hundreds of rows) usually go to:

  * Appendix
  * External data files (referenced in the thesis)
* In the main chapters, include only:

  * Representative subsets
  * Summaries and statistics

## 3. Text data (documents, transcripts, logs, code)

### 3.1 Preparing text data

**File formats**

* Use plain text formats when possible:

  * `.txt`, `.csv`, `.json`, `.xml`, `.md`
* For transcriptions or combined information:

  * `.csv` with columns like `id`, `speaker`, `timestamp_start`, `timestamp_end`, `text`
* For code:

  * Language-specific files (`.py`, `.java`, `.cpp`)
  * Use a consistent style and linter.

**Cleaning & normalization**

* Decide what “cleaning” means for your project and document it:

  * Lowercasing
  * Removing punctuation
  * Removing stopwords (for NLP tasks)
  * Normalizing whitespace
* If you keep original vs. cleaned text, store separately:

  * `data/text/raw/`
  * `data/text/cleaned/`

**Segmentation**

* Define your unit of analysis:

  * Document, paragraph, sentence, message, log entry.
* Use IDs for each unit:

  * `doc_id`, `sentence_id` etc.

**Annotations / labels**

* If you label text (e.g. sentiment, category), document:

  * The label set (e.g. `positive`, `neutral`, `negative`)
  * Annotation guidelines: how did you decide each label?
  * Who annotated and how many annotators?

### 3.2 Using text data in the thesis

**Quotations**

* Short text examples:

  * Put inside quotation marks in the text.
* Longer excerpts:

  * Use a block quote or a separate figure/table.
* Always indicate:

  * Source (dataset, author, URL if external)
  * Whether the text was translated or edited.

**Describing preprocessing**

* In the methodology chapter, explain:

  * What transformations were applied (e.g. tokenization, stemming).
  * Which tools or libraries you used (e.g. NLTK, spaCy).
* Example description:

  > “All texts were converted to lowercase, and punctuation was removed. Stopwords from the English NLTK list were filtered out. Finally, words were lemmatized using spaCy’s English model.”

**Code snippets**

* Keep code readable and short in the main text:

  * Use syntax highlighting if possible.
  * For longer scripts, put them in an appendix or provide as supplementary files.
* Describe what the code does, not only show it.

## 4. Image data (photos, diagrams, plots, scans)

### 4.1 Preparing image data

**File formats**

* For raster images:

  * `.png` (lossless; good for plots, diagrams)
  * `.jpg` / `.jpeg` (compressed; OK for photos)
* For vector graphics (best for line art / diagrams):

  * `.svg`, `.pdf`, `.eps`
* Keep original high-resolution files.

**Resolution & quality**

* Use sufficiently high resolution for printing:

  * Typically at least 300 dpi for print.
* Avoid heavy compression artifacts (no low-quality JPEG).

**Naming & organization**

* Use descriptive names:

  * `exp01_setup_photo_01.jpg`
  * `exp02_results_plot_accuracy.png`
* Keep raw and edited versions separate:

  * `images/raw/`
  * `images/processed/`

**Metadata**

* If relevant, store:

  * Camera model, focal length, location, date.
* For generated plots, document:

  * Data source
  * Parameters (axis scales, filters, thresholds used to create the image).

**Preprocessing**

* When you edit images (crop, adjust brightness, add annotations):

  * Keep a copy of the original.
  * List the modifications (especially important in scientific images):

    * “Cropped to region of interest.”
    * “Contrast enhanced uniformly.”
  * Do **not** manipulate images in a way that changes the meaning of the data.

### 4.2 Using images in the thesis

**Captions and numbering**

* Each image should appear as a **Figure**:

  * “Figure 3.4: Schematic diagram of the test stand.”
* Caption must briefly explain:

  * What is shown
  * Possibly key elements or color coding

**Readability**

* Make sure labels inside the figure are readable after printing:

  * Use sufficiently large font sizes in plots.
  * Avoid excessive text inside the image; use the caption for long explanations.

**Referencing in text**

* Example:

  * “The overall test stand is shown in Figure 3.4.”
  * “Figure 5.2 illustrates the relationship between speed and torque.”

**External images**

* If you use an image not created by you:

  * Include citation in the caption:

    * “Source: [Author, Year]” or “Source: [Online resource]”.
  * Make sure you have the right to reproduce the image.

## 5. Audio data (recordings, speech, signals)

### 5.1 Preparing audio data

**File formats**

* Use lossless formats for analysis:

  * `.wav` (preferred)
  * Sample rate and bit depth: e.g. `16 kHz`, `16-bit` or `44.1 kHz`, `16-bit`
* Compressed formats for distribution:

  * `.mp3`, `.ogg` (OK for sharing, but not ideal for signal processing)

**Recording setup documentation**

* Record:

  * Device and microphone model
  * Sample rate and bit depth
  * Environment (room type, noise level)
  * Microphone distance from source
  * Recording conditions (time, speaker, etc.)

**File naming**

* Include content, speaker, and version in the file name:

  * `speaker01_sentence10_clean.wav`
  * `test_signal_sine_1kHz_44k1.wav`

**Metadata**

* Create a table (CSV) describing each file with columns such as:

  * `file_name`
  * `speaker_id`
  * `language`
  * `duration_seconds`
  * `sampling_rate_hz`
  * `label` (e.g. class, emotion, word)
  * `notes` (noise, errors, etc.)

**Preprocessing**

* Typical steps:

  * Normalization (volume)
  * Noise reduction (document methods used)
  * Trimming silence
  * Resampling to a standard sample rate
* Save processed audio separately from raw:

  * `audio/raw/`
  * `audio/processed/`

### 5.2 Using audio data in the thesis

**Description in the methodology**

* Explain:

  * How you recorded/obtained the audio
  * The recording setup and parameters
  * Any preprocessing steps
* Example:

  > “Audio was recorded at a sampling rate of 16 kHz and 16-bit resolution using a cardioid condenser microphone at a distance of approximately 20 cm. Background noise was reduced using spectral subtraction.”

**Transcriptions**

* For speech data:

  * Provide written transcriptions for important examples.
  * Indicate transcription conventions (e.g. marking pauses, overlaps).
* Include samples of transcripts in tables or appendices.

**Referring to audio examples**

* If your university allows supplementary files:

  * Mention them in the text:

    * “Example recordings are provided in the supplementary material (files `audio_example_01.wav`, `audio_example_02.wav`).”
* Describe what they illustrate:

  * “This example demonstrates the noise level before filtering.”

## 6. Video data (experiments, simulations, demonstrations)

### 6.1 Preparing video data

**File formats**

* Common containers:

  * `.mp4` (most practical)
  * `.avi` (less compressed, larger size)
* Choose a codec with reasonable quality and compatibility.

**Resolution & frame rate**

* Decide on standard settings:

  * e.g. `1920x1080`, `30 fps`
* Use consistent resolution and FPS across videos when possible.
* Document them in a metadata file.

**File naming**

* Examples:

  * `exp01_robot_arm_motion_trial01.mp4`
  * `simulation_fluid_flow_caseA.mp4`

**Metadata**

* Create a catalog (CSV or table) including:

  * `file_name`
  * `description`
  * `duration_seconds`
  * `resolution`
  * `fps`
  * `date_recorded`
  * `camera` or software used
  * `scenario` / `experiment_id`

**Editing & preprocessing**

* If you cut, speed up, or modify videos:

  * Keep original raw recordings.
  * Document edits:

    * “Clipped to first 10 seconds.”
    * “Playback speed ×2.”
    * “Overlaid annotations (labels for objects).”
* Avoid modifications that could misrepresent the experiment.

### 6.2 Using video data in the thesis

**Still frames as figures**

* Often you cannot embed full videos in printed theses, so:

  * Extract key frames or screenshots.
  * Use them as figures with captions explaining what moment they show.
* Mention associated video files:

  * “Figure 6.3 shows a key frame from the trajectory demonstration (see video file `exp05_trajectory_demo.mp4`).”

**Description of content**

* Describe:

  * What the video shows
  * Experimental conditions
  * Key events or timestamps
* Example:

  > “The video recording shows the robot arm executing a pick-and-place task at a speed of 0.5 m/s. At 00:07, the gripper fails to grasp the object, illustrating the effect of positioning error.”

**Supplementary material**

* Place videos in a `media/video/` directory.
* Reference them in the thesis:

  * “Video files with full experiment recordings are provided as supplementary material.”

## 7. Putting it all together — structure for your engineering thesis

You can systematically describe your data and preparation process in one chapter (often “Materials and Methods” or “Data and Methods”):

1. **Data sources**

   * Describe all datasets:

     * Tables (sensor data, logs)
     * Text (documents, transcripts)
     * Images, audio, video
   * For each, specify:

     * Origin (own measurements, public dataset, partner company)
     * Time period, location, equipment, software

2. **Data formats and storage**

   * Describe folder structure and naming conventions briefly.
   * Mention main file formats (CSV, WAV, MP4, etc.).

3. **Preprocessing and cleaning**

   * For each data type:

     * What preprocessing steps were applied
     * What tools and parameters you used
     * How you handled missing or noisy data

4. **Annotations and labels**

   * How labels were defined and assigned (for supervised tasks).
   * Inter-annotator agreement if multiple people annotated.

5. **Ethical and legal considerations**

   * Data anonymization steps, if applicable.
   * Licensing of external datasets.

6. **Presentation of results**

   * Explain how you present data:

     * Tables for numeric summaries
     * Figures for key plots and images
     * Example texts/audio/video for qualitative illustration
   * Ensure every element is referenced and interpreted.
