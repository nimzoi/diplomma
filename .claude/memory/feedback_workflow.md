**Speed-run mode acknowledged.** Magda pracuje burstami ("6 miesięcy w 3 dni"). Brak kalendarzowych deadline'ów per iteracja — kadencja diktowana intensity, nie zegarem.

**ALE done criterion każdej iteracji jest TWARDY.** Iteracja N+1 startuje gdy N done criterion met. Re-iteration (2a, 2b) acceptable w razie failure.

**Why:** żeby utrzymać dyplom przy speed-run mode konieczne jest decoupling kadencji od kalendarza — inaczej self-imposed deadlines kosztują motywację. Twarde done criteria są jedynym hard contract z samą sobą.

**How to apply:**
- **NIE proponować "weekly check-in", "monthly sync", "deadline za 2 tygodnie".** Iteracja jest jednostką pracy, nie tydzień.
- **Proponować trade-offy:** "Iteracja X może być compressed (np. 1500 docs zamiast 4100), defensible scope ale flagged w R8 limitations" — speed-run accepts to.
- **Zawsze pytaj o done criterion** przed startem iteracji — jeśli nie ma jasnego "skąd wiem że to gotowe", iteracja nie startuje.
- **Promotor dostaje artefakty async** (DVC pull / repo state) — nie scheduled email. Nie proponować "wyślij update do promotora w piątek".
- **Track-parallel strategy** (writing track + experimental track + cleanup track) jest preferowane nad sequential w speed-run mode.
