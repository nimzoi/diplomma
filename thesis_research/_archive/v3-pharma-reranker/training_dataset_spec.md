# Training Dataset Specification — pharma reranker fine-tuning

**Status:** ACTIVE (2026-05-16)
**Domena:** farmakologia kliniczna (psych eval subset N05/N06)
**Cel:** dataset preferencji dla fine-tuningu `polish-reranker-roberta-v3` w 3 cyklach
**Single source of truth for:** R4 (EDA preprocessing), R6 (modele training), R7 (results — dataset characterization)
**Related:** [`sources_catalog.md`](sources_catalog.md), [`02b_konspekt_v3_updates.md`](02b_konspekt_v3_updates.md) § II.4.6 (hard negatives), [`decisions/DEC-002_chpl-ulotka-pairing.md`](decisions/DEC-002_chpl-ulotka-pairing.md) (cross-register RQ5)

---

## Overview

Reranker dotrenowywany w 3 cyklach iteracyjnego retreningu. W każdym cyklu generujemy **preference quadruplets** w formacie:

```
(query, positive_passage, [negative_1, negative_2, negative_3])
```

z **multi-level hard negative sampling strategy** (II.4.6 → 4 poziomy difficulty).

**Estymowany finalny rozmiar datasetu (cykl 3):** ~145k preference samples po `cumulative growth + dedup`. Per-cycle approx 50k samples.

**Generator:** Bielik 11B v3 (Apache 2.0) jako primary, optional fallback to `<judge_model>` z Iteracji 0b (Gemma 3 27B / Qwen 3 32B / Claude Haiku 4.5).

---

## 1. Synthetic query generation pipeline

Reranker training wymaga queries — corpus sam zawiera passages. Strategia: **back-generation** queries z passages przez LLM (InPars-style methodology, Bonifacio et al. 2022).

### 1.1 Generator model selection

| Option | Pros | Cons | Use |
|---|---|---|---|
| **Bielik 11B v3** | Polish native, Apache 2.0, on-prem | Smaller model — niższa diversity | **Primary** (70%) |
| `<judge_model>` (z Iter. 0b) | Cross-validation (different model than judge) | Może być Claude API (koszt) | **Fallback** dla edge cases (30%) |
| PLLuM-12B (alt) | Polish native, larger | CC-BY-NC-4.0 limits | Alt if Bielik underperforms |

**Rationale dla Bielik primary:** generator i judge powinny być **różne modele** żeby uniknąć circular reasoning (jeśli judge ocenia preferences generated by same model, agreement będzie sztuczna wysoka).

### 1.2 Query types taxonomy (6 typów, mapowane na sekcje ChPL/Ulotka)

| # | Query type | Przykład | Gold passage source |
|---|---|---|---|
| 1 | **Factual / Lookup** | *„Jakie są przeciwwskazania do sertraliny?"* | ChPL sekcja 4.3 |
| 2 | **Procedural** | *„Jak przyjmować lek X w niewydolności wątroby?"* | ChPL sekcja 4.2 |
| 3 | **Comparative** | *„Czy sertralina i fluoksetyna mogą być przyjmowane razem?"* | ChPL sekcja 4.5 (Interakcje) |
| 4 | **Edge case / Scenario** | *„Co zrobić, gdy zapomniałem dawki?"* | Ulotka sekcja 3 |
| 5 | **Definitional** | *„Co to jest lek X?"* | Ulotka sekcja 1 lub ChPL sekcja 5 (Farmakologia) |
| 6 | **Adverse effects** | *„Jakie są skutki uboczne X?"* | ChPL sekcja 4.8 / Ulotka sekcja 4 |

**Polish-specific augmentation w obrębie każdego typu:** dla każdego base query generujemy 1-2 **grammatical inflection variants** (różne polskie przypadki + alternative word order) — uchwycenie polish-specific patterns które są właściwym motywatorem domain adaptation (zobacz 02b II.2.1 pkt 1; DCI sama jest reprezentowana w pretrainingu, ale polska fleksja DCI niekoniecznie). Przykład per Factual:

- Base: *„Jakie są przeciwwskazania do sertraliny?"* (Genitive)
- Variant 1: *„Czy sertralina ma przeciwwskazania?"* (Nominative + framing change)
- Variant 2: *„Przeciwwskazania dla sertraliny — jakie?"* (Reorder + Dative)

Wszystkie warianty mapują na **ten sam positive passage** (ChPL sekcja 4.3 sertraliny) — uczą reranker fleksji + word order independence dla zachowanego semantic intent. Multiplikator ~1.5× base query count (np. 30k base factual → ~45k z variants).

Generation strategy: po wygenerowaniu base query, dodatkowy prompt LLM: *„Sparafrazuj to pytanie używając innego przypadka gramatycznego lub innego szyku wyrazów, zachowując semantic intent"*. Walidacja: spot-check że variant query nadal ma ten sam positive passage (semantic equivalence).

### 1.3 Proporcje per query type

Target distribution w training set (per cycle):

| Type | Proportion | Cycle 1 count | Rationale |
|---|---|---|---|
| Factual | 30% | ~15k | Most common real query type |
| Procedural | 20% | ~10k | Common w clinical decision support |
| Adverse effects | 20% | ~10k | High-value w pharmacy domain |
| Definitional | 10% | ~5k | Easier — anchor type |
| Edge case | 10% | ~5k | Real-world patient query type, helps RQ5 cross-register |
| Comparative | 10% | ~5k | Hardest type — saved for diversity (i.e. cross-section comprehension) |

**Total target cycle 1:** ~50k queries × avg 3 negatives = 200k records. Po dedup + quality filter → ~50k positive pairs + ~150k preference samples.

### 1.4 Prompt template (few-shot)

```text
[INSTRUCTION]
Wygeneruj pytanie po polsku, na które poniższy fragment dokumentu farmaceutycznego stanowi pełną odpowiedź. Pytanie ma być w stylu {QUERY_TYPE} (e.g., "factual lookup", "procedural", "comparative", etc.). Pytanie powinno być naturalne, jakby zadał je {USER_PROFILE} (e.g., "lekarz", "farmaceuta", "pacjent").

[FEW-SHOT EXAMPLE 1]
Document type: ChPL sekcja 4.3 Przeciwwskazania
Passage: "Nadwrażliwość na sertraliną lub którąkolwiek substancję pomocniczą. Jednoczesne stosowanie z inhibitorami MAO..."
User profile: lekarz
Query type: factual
Generated query: "Jakie są bezwzględne przeciwwskazania do stosowania sertraliny?"

[FEW-SHOT EXAMPLE 2]
Document type: Ulotka sekcja 3 Jak stosować
Passage: "Jeśli zapomnisz przyjąć dawkę leku, przyjmij ją jak najszybciej, ale jeśli zbliża się pora następnej dawki..."
User profile: pacjent
Query type: edge case
Generated query: "Co zrobić, gdy zapomnę dawki leku?"

[TASK]
Document type: {SECTION_HEADER}
Passage: {PASSAGE_TEXT}
User profile: {USER_PROFILE}
Query type: {QUERY_TYPE}
Generated query:
```

**Validation:** każdy wygenerowany query passes through 3-stage filter:
1. Length check (5-50 słów)
2. Polish language detection (`langdetect` + manual heuristic na zakazane ang. słowa)
3. Question form check (kończy się `?` lub zawiera pytajnik)

Reject rate target: ~10-15% (rejected → discard).

### 1.5 Quality validation

**Spot-check methodology:** manual review 100 random generated queries per cycle przez autorkę.

Quality criteria:
- ✅ Query is grammatically correct Polish
- ✅ Query is answerable by the passage (semantic alignment)
- ✅ Query is realistic — user could plausibly ask this
- ❌ NIE jest re-statement passage'a (e.g., "Co mówi sekcja 4.3?")
- ❌ NIE wymaga external knowledge spoza passage'a

**Acceptance threshold:** ≥80% of spot-checked queries muszą być valid. Poniżej — prompt template revision.

## 2. Positive sampling (query → gold passage)

Dla każdego wygenerowanego query, gold passage jest **passage'em z którego query został wygenerowany**. To deterministyczny alignment.

### 2.1 Per-strata sampling strategy

| Strata | Positive selection | Sample target |
|---|---|---|
| **ChPL** | Section-aware split (10 sekcji) → 1 query per section per drug → 900 leków × 9 sections × ~6 queries = ~48k positives | 50% |
| **Ulotki** | 6 sekcji QRD → 1 query per section per drug → 900 × 6 × ~5 = ~27k positives | 28% |
| **AOTMiT + MZ HTA** | Named sections (Problem decyzyjny / Skuteczność / Bezpieczeństwo / Analiza ekonomiczna) → ~5k positives | 5% |
| **NFZ Zarządzenia** | B.xx program sections (Kryteria włączenia / Schemat dawkowania / Monitorowanie) → ~2k positives | 2% |
| **OA journals** | Section-based (Wstęp / Metody / Wyniki / Dyskusja) → 1-2 queries per article → ~12k positives | 12% |
| **Adjacencies (DHPC + braki)** | Whole document as positive (short docs) → ~1.5k positives | 1.5% |
| ~~Synthetic paraphrases~~ | Optional augmentation w cyklu 2+ | flag w R6 |
| **Total** | | **~95k positives base** |

Pamiętaj: każdy positive paired z 3 negatives = ~285k preference samples raw, po dedup + quality filter → ~145k.

### 2.2 Cross-register positives (RQ5-specific)

Dla **RQ5 cross-register evaluation**, dodatkowo generujemy 1800 specific pairs:

- **lay→pro (900 par):** query z Ulotki sekcji → gold = ChPL sekcji tego samego leku, semantycznie równoważnej
- **pro→lay (900 par):** odwrotnie

**Alignment mechanism:** deterministyczny przez `productID` z URPL RPL feed + section semantic mapping (zdefiniowane w `sources_catalog.md` § Strata 2 "Definicja paired ChPL↔Ulotka").

Mapping sekcji (manual table):

| Ulotka sekcja | ChPL sekcja odpowiadająca |
|---|---|
| 1. Co to jest lek X | 1. Nazwa + 5. Farmakologia (parts) |
| 2. Informacje ważne przed przyjęciem | 4.3 Przeciwwskazania + 4.4 Ostrzeżenia |
| 3. Jak stosować | 4.2 Dawkowanie |
| 4. Możliwe działania niepożądane | 4.8 Działania niepożądane |
| 5. Jak przechowywać | 6.4 Specjalne środki ostrożności przechowywania |
| 6. Zawartość opakowania | 6.5 Rodzaj i zawartość opakowania |

## 3. Negative sampling — 4-level strategy

**Reference:** `02b_konspekt_v3_updates.md` § II.4.6 (Hard negatives strategy explicit). NIE duplikujemy tu.

**Skrót:**

| Level | Strategy | Proportion w 3 negatives per quadruplet | Difficulty |
|---|---|---|---|
| **L1 (easy)** | Different ATC class | 15% | 🟢 |
| **L2 (medium)** | Same ATC class, different drug | 50% | 🟡 |
| **L3 (hard)** | Same drug, different section | 30% | 🔴 |
| **L4 (very hard, RQ5)** | Same drug, cross-register confusion | 5% | 🔴🔴 |

Każdy positive → 3 hard negatives próbkowane z proportional levels (L2 dominuje).

### 3.1 Hard negative mining methodology

Dla każdego query Q with positive P:

1. **Compute embedding similarity** sim(Q, P) via BGE-M3
2. **Candidate pool:** wszystkie passages z corpus
3. **Filter** by ATC class / drug / section per level definition
4. **Score similarity** sim(Q, candidate) dla każdego candidate w filtered pool
5. **Hard negative** = candidate with sim(Q, candidate) > threshold (np. 0.6) ale NIE positive

Threshold tuning per cycle:
- Cykl 1: threshold = 0.5 (looser → easier negatives → smooth start)
- Cykl 2-3: threshold = 0.6-0.7 (harder negatives → fine-grained discrimination)

## 4. Final training data format

### 4.1 Decision: quadruplet, NIE triplet, NIE listwise

| Format | Pros | Cons | Decision |
|---|---|---|---|
| Triplet (q, p, n) | Simpler, more samples | Less discriminative power | ❌ |
| **Quadruplet (q, p, n1, n2, n3)** | **Multi-level negatives w jednym sample, listwise-ish** | More compute per step | ✅ **chosen** |
| Listwise (q, ranked list) | Most discriminative | Hardest to generate, slow | ❌ — overkill dla ~360M model |

**Reasoning:** quadruplet pozwala model uczyć się **simultaneously** rozróżniać L1/L2/L3 negatives w obrębie jednego training step. Triplet zmusza separate steps per level. Listwise wymaga full ranking — too expensive.

### 4.2 File format: JSONL

```jsonl
{"query": "Jakie są przeciwwskazania do sertraliny?", "positive": "Nadwrażliwość na sertralinę lub którąkolwiek substancję pomocniczą...", "negatives": ["Sertralina jest wskazana w leczeniu epizodów dużej depresji...", "Fluoksetyna jest przeciwwskazana u pacjentów z...", "Sertralina może wpływać na zdolność prowadzenia pojazdów..."], "metadata": {"productID": "39257", "atc": "N06AB06", "positive_section": "4.3", "query_type": "factual", "negative_levels": ["L3", "L2", "L3"], "cycle": 1, "generation_seed": 42, "generator_model": "bielik-11b-v3"}}
```

### 4.3 Per-record fields

| Field | Type | Description |
|---|---|---|
| `query` | string | Synthetic query (Polish, ≤50 words) |
| `positive` | string | Gold passage (relevant) |
| `negatives` | list[string] (length 3) | Hard negatives, mixed levels |
| `metadata.productID` | string | URPL productID for positive's source drug |
| `metadata.atc` | string | ATC code (e.g., "N06AB06") |
| `metadata.positive_section` | string | Source section (e.g., "ChPL 4.3", "Ulotka 3") |
| `metadata.query_type` | enum | One of: factual / procedural / comparative / edge_case / definitional / adverse_effects |
| `metadata.negative_levels` | list[string] (length 3) | Per-negative level: L1 / L2 / L3 / L4 |
| `metadata.cycle` | int | Iteration cycle (1, 2, lub 3) |
| `metadata.generation_seed` | int | Random seed dla reproducibility |
| `metadata.generator_model` | string | Model name (e.g., "bielik-11b-v3") |

## 7. Validation methodology

### 7.1 Pre-training validation (per cycle, 200 samples random)

Autorka manualnie reviewuje 200 randomly sampled preference quadruplets z każdego cyklu:

| Sprawdzenie | Acceptance threshold |
|---|---|
| Query is grammatically correct PL | ≥95% |
| Query is semantically aligned z positive | ≥90% |
| Each negative is genuinely worse than positive (NIE tied) | ≥85% |
| Negative levels (L1-L4) zgodne z target proportions | ±5% tolerance |
| Brak duplicate queries within cycle | <1% duplication rate |

**Acceptance pipeline pass criteria:** wszystkie ≥ threshold. Failure → prompt template revision lub sampling threshold tuning, re-generate cycle.

### 7.2 Cross-cycle consistency (post-cycle 3)

Sprawdzić że dataset growth nie deformuje query type distribution:
- Cycle 1 → cycle 3: każdy query type powinien zachować ±5% proportion
- ATC class distribution: zachowany 30% N05/N06 over-rep z `sources_catalog.md` sampling algorithm

## 8. Świadome biases / limitations (do R3 + R8)

1. **LLM-generated queries bias:** queries są synthetic, NIE real user queries. Generator model (Bielik) ma swoje patterns — możliwe że queries są bardziej formalne / mniej noisy niż real user queries.
   - **Mitigation:** User profile prompt diversity (lekarz / farmaceuta / pacjent / student), query type proportions calibrated to realistic distribution.
2. **Positive section alignment bias:** assumption że ChPL section header → body to natural query→answer. To IS deterministic, ale może over-fit reranker na exact section structure.
   - **Mitigation:** synthetic queries paraphrased zamiast literal section headers; cross-register pairs (RQ5) zmuszają model do generalize beyond section structure.
3. **Hard negative quality:** L2/L3 negatives generowane przez similarity threshold — potencjalnie tied (positive ≈ negative semantically) jeśli threshold za wysoki.
   - **Mitigation:** spot-check 100 quadruplets per cycle, manually flag tied; adjust threshold.
4. **Query type proportions arbitrary:** 30% factual / 20% procedural / etc. — bazowane na intuicji o pharma user patterns. Brak empirycznego źródła dla tych proportions w PL.
   - **Mitigation:** flag explicit w R8 limitations; future work: analyze real query logs (jeśli kiedyś deployed).

## 9. Anti-patterns (do unikania)

- ❌ **Real user queries** scraped z forów pacjentów (etyka, prawa autorskie, RODO)
- ❌ **Translated EN queries** (BEIR/MS MARCO PL translation) — mieszanie z native PL może zaśmiecić
- ❌ **Self-judge:** generator (Bielik) jednocześnie jako judge (`<judge_model>`) — circular reasoning, sztuczna wysoka kappa
- ❌ **Random negatives bez stratification** — model uczy się coarse separation, nie fine-grained discrimination
- ❌ **Listwise format** dla ~360M model — overkill, slow training
- ❌ **No spot-check** between cycles — silent degradation if generator drifts
